    База данных¶

    Основные понятия¶

    1. База данных (БД) — это организованная совокупность данных, предназначенная для длительного хранения во внешней памяти и постоянного использования. Это структурированное хранилище информации, которым управляет специальное программное обеспечение.

    2. Система управления базами данных (СУБД) — комплекс программных средств, обеспечивающих создание, ведение и совместное использование баз данных. Примеры: PostgreSQL, Oracle, MongoDB.

    3. Модели данных

      Способ организации данных в БД определяется моделью данных. Наиболее распространены:

      • Реляционная модель — данные представляются в виде таблиц (отношений), связанных между собой.

      • Нереляционные модели (NoSQL) — включают документные, графовые, ключ-значение и колоночные модели.

    Реляционные базы данных (SQL)¶

    Реляционные БД используют табличное представление и язык структурированных запросов SQL.

    1. Ключевые понятия реляционной модели

      Термин Описание Пример из документации
      Таблица (relation) Совокупность строк и столбцов, хранящая данные об объектах одного типа Таблица users содержит информацию о пользователях
      Строка (row, запись, кортеж) Один экземпляр объекта, набор значений по всем столбцам Каждая строка в таблице users — это отдельный пользователь
      Столбец (column, поле, атрибут) Хранит однотипные характеристики объектов Столбец email содержит адреса электронной почты
      Первичный ключ (Primary Key, PK) Уникальный идентификатор строки (одно или несколько полей). Не может быть NULL user_id — первичный ключ таблицы users
      Внешний ключ (Foreign Key, FK) Поле, ссылающееся на первичный ключ другой таблицы, для установления связи В таблице orders поле user_id — внешний ключ к таблице users

    2. Нормализация

      Процесс организации данных для устранения избыточности и аномалий. Обычно выделяют несколько нормальных форм (1НФ, 2НФ, 3НФ). При описании структур данных полезно указывать степень нормализации, если это важно для понимания.

    3. Язык SQL

      SQL (Structured Query Language) — стандартный язык для работы с реляционными БД. Чтобы правильно описывать функциональность базы данных важно понимать основные группы операторов:

      • DDL (Data Definition Language) — создание и изменение структуры: CREATE, ALTER, DROP.

      • DML (Data Manipulation Language) — работа с данными: INSERT, UPDATE, DELETE, SELECT.

      • DCL (Data Control Language) — управление доступом: GRANT, REVOKE.

      • TCL (Transaction Control Language) — управление транзакциями: COMMIT, ROLLBACK.

    4. Пример описания реляционной структуры

      В документации часто приводятся схемы таблиц. Хороший тон — описывать каждую таблицу и ее поля с указанием типа данных, ограничений и назначения.

      Пример (таблица employees):

      Поле Тип Ограничения Описание
      employee_id INTEGER PRIMARY KEY, AUTO_INCREMENT Уникальный идентификатор сотрудника
      first_name VARCHAR(50) NOT NULL Имя сотрудника
      last_name VARCHAR(50) NOT NULL Фамилия сотрудника
      hire_date DATE Дата найма
      department_id INTEGER FOREIGN KEY (departments) Идентификатор отдела, в котором работает сотрудник

    Нереляционные базы данных (NoSQL)¶

    NoSQL-базы используются, когда требуется высокая производительность, масштабируемость или гибкая схема данных. Основные типы:

    1. Документные базы (MongoDB).

      Хранят данные в виде документов (JSON, BSON). Каждый документ может иметь собственную структуру.

    2. Базы типа «ключ-значение» (Redis, DynamoDB).

      Данные хранятся как пары «ключ — значение». Очень быстрые, используются для кэширования, сессий.

    3. Колоночные базы (Cassandra, HBase).

      Оптимизированы для хранения больших объёмов данных и аналитических запросов. Данные организованы по столбцам, а не по строкам.

    4. Графовые базы (Neo4j, Amazon Neptune).

      Используются для данных со сложными связями (социальные сети, рекомендации). Сущности — узлы, связи — ребра.

    Критерии выбора типа БД¶

    При описании архитектуры продукта полезно пояснять, почему выбрана та или иная СУБД. Общие критерии:

    • Структурированность данных — если данные строго типизированы и связаны, чаще используют SQL.

    • Объём и нагрузка — для высоких нагрузок и горизонтального масштабирования часто выбирают NoSQL.

    • Требования к согласованности (ACID или BASE) — для финансовых операций критична согласованность (ACID → SQL), для многих веб-приложений допустима конечная согласованность (BASE → NoSQL).

    • Тип запросов — сложные аналитические отчеты либо простые операции по ключу.

    Рекомендации по описанию баз данных в документации¶

    1. Для пользовательской документации (Руководства пользователя):

      • Избегайте излишней технической детализации (например, индексы, планы запросов).

      • Объясняйте, какие данные сохраняются, как к ним обратиться, какие ограничения существуют (например, максимальная длина поля).

      • Если продукт позволяет выполнять запросы, покажите примеры с пояснениями.

    2. Для административной документации (Руководство системного программиста):

      • Детально описывайте структуру БД: таблицы, поля, типы данных, индексы, ограничения.

      • Приводите схемы связей (ER-диаграммы).

      • Описывайте процедуры резервного копирования, восстановления, мониторинга.

      • Объясняйте параметры конфигурации, влияющие на производительность.

    3. Общие правила:

      • Используйте единую терминологию во всей документации.

      • Добавляйте глоссарий терминов.

      • Сопровождайте описания примерами данных и запросов.

      • Для наглядности используйте диаграммы (ER-диаграммы для SQL, схемы документов для NoSQL).

    Глоссарий терминов¶

    Термин Определение
    Атрибут Свойство или характеристика сущности (объекта), которая описывается в таблице базы данных
    Индекс Структура, ускоряющая поиск по определённым полям
    Ключ Поле или набор полей, используемых для идентификации строки (первичный ключ) или связи (внешний ключ)
    Миграция Процесс изменения схемы БД (например, добавление столбца) с сохранением данных
    Нормализация Устранение избыточности данных путем разбиения на связанные таблицы
    Репликация Копирование данных на несколько серверов для надежности и распределения нагрузки
    Транзакция Логически связанная последовательность операций, которые выполняются как единое целое
    Шардинг Горизонтальное разделение данных по разным серверам (часто используется в NoSQL)
    ER-диаграмма Диаграмма «сущность-связь», отображающая таблицы и отношения между ними
    Коллекция (в MongoDB) Аналог таблицы в документных БД, хранит набор документов
    Документ Запись в документной БД, обычно в JSON-формате
